## Priority and traceability rules

- **P0:** required for the implementation-complete single-node v0.1 release candidate and Build Week submission.
- **P1:** useful follow-up quality or operability work that is not required for the v0.1 release decision.
- **P2:** post-v0.1 optional log-adapter or roadmap work; it is excluded from every release gate and must not become a prerequisite of P0 work.
- Each milestone exit criterion applies to that milestone's **P0 tasks**. A P1 task placed earlier may be completed during stabilization after its own prerequisites are met.
- **Layer separation:** backend/contract implementation and frontend/UI/UX implementation always use separate Task IDs, deliverables, completion criteria, and evidence. A later integration task may verify both but must not absorb unfinished implementation from either layer.
- **Traceability:** each row references PRD `FR-*`/`NFR-*`, ADR `ADR-*`, and TDD test or design labels.
- **Canonical TDD test IDs:** unit `UT-001~019`, contract `CT-001~010`, integration `IT-001~015`, end-to-end `E2E-001~012`, security `SEC-001~014`, and recovery `REC-001~010`. Labels such as `VAL-*` and `SIM-*` are navigation aliases; TDD owns the final mapping.
- **Verification evidence:** retain applicable test output, API response, audit event, screenshot, or reproduction command in the PR or release record.

## Safety-path invariants

The README enforcement path is implemented as ordered, non-bypassable hard gates:

```text
AI evidence-bound policy and nftables blacklist command candidate
  → ① Structured-output/schema and command parsing/canonicalization
  → ② Policy-evidence-command consistency
  → ③ Protected-target check
  → ④ nftables syntax validation against the owned-set schema
  → ⑤ Historical-impact analysis
  → ⑥ Administrator HIL approval of the exact artifact by digest
  → ⑦ Shell-free temporary application in isolation
  → ⑧ Automatic expiry or manual revocation
```

- Audit start, success, failure, rejection, and expiry for each stage.
- If any result fails or is missing, do not enter the next stage; fail closed.
- Bind approval to `incident_id`, policy version/digest, generated/canonical command digests, immutable evidence/validation snapshot digest, actor, reason, and the validation/decision validity windows. Any dependent policy, command, evidence, analysis, parser, validator, configuration, or validation change invalidates approval.
- v0.1 enforcement is limited to an isolated container or network namespace. Host and production firewall enforcement is disabled by default and excluded from P0 completion.
- Every applied rule has a TTL. New `execution-result-v2` terminal evidence signs the read-back interval; lifecycle retains lower/upper expiry bounds and never infers one exact kernel expiry from a broad execution interval. After restart or failed expiry, recovery uses the two-phase journal and read-back to classify state, may remove expired or unapproved residue, and never re-adds a missing rule, reuses the source bound, or refreshes its relative TTL. Any future add requires a new candidate, validation snapshot, exact-artifact HIL decision, and single-use capability.
- The Gateway is an unprivileged asynchronous data plane in front of one fixed private upstream. It has no `CAP_NET_ADMIN`, never waits for AI or the control plane, and forwards during control-plane outage without creating or applying a new block.
