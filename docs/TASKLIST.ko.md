## 우선순위와 추적 규칙

- **P0**: 구현 완료된 단일 노드 v0.1 릴리스 후보와 Build Week 제출에 필수인 작업.
- **P1**: v0.1 릴리스 판정에는 필요하지 않은 후속 품질·운영성 작업.
- **P2**: v0.1 이후의 선택형 로그 어댑터 또는 로드맵 작업. 모든 릴리스 게이트에서 제외하고 P0 작업의 선행조건으로 삼지 않는다.
- 각 마일스톤의 종료 조건은 그 마일스톤의 **P0 작업**에 적용한다. 앞 마일스톤에 배치된 P1 작업은 표에 적힌 별도 선행조건을 만족한 뒤 안정화 단계에서 수행한다.
- **계층 분리**: 백엔드/계약 구현과 프론트엔드/UI·UX 구현은 항상 서로 다른 Task ID, 산출물, 완료 기준과 검증 증거로 관리한다. 후속 통합 작업은 양쪽을 함께 검증할 수 있지만 어느 한 계층의 미완료 구현을 흡수해서는 안 된다.
- **추적**: 각 행의 `관련 문서 / 요구사항`은 [PRD](./PRD.ko.md)의 `FR-*`, `NFR-*`, [ADR](./ADR.ko.md)의 `ADR-*`, [TDD](./TDD.ko.md)의 테스트 범주를 가리킨다.
- **TDD 공식 ID**: 단위 `UT-001~019`, 계약 `CT-001~010`, 통합 `IT-001~015`, 종단 간 `E2E-001~012`, 보안 `SEC-001~014`, 복구 `REC-001~010`을 사용한다. 행에 병기한 `VAL-*`, `SIM-*` 같은 의미 레이블은 해당 공식 테스트 ID의 목적을 빠르게 찾기 위한 별칭이며 최종 교차표는 TDD를 기준으로 한다.

## 안전 경로 불변식

README가 요구하는 집행 경로는 다음 순서의 **우회 불가능한 하드 게이트**로 구현한다.

```text
AI 근거 기반 policy 및 nftables blacklist command candidate
  → ① Structured-output/schema 및 command parsing/canonicalization
  → ② Policy-evidence-command consistency
  → ③ Protected-target 검사
  → ④ Owned-set schema 대상 nftables syntax validation
  → ⑤ Historical-impact analysis
  → ⑥ Exact artifact의 digest 기준 관리자 HIL 승인
  → ⑦ 격리 환경에 shell-free 임시 적용
  → ⑧ 자동 만료 또는 수동 해제
```

- 각 단계의 시작·성공·실패·거부·만료를 감사 로그에 남긴다.
- 한 단계라도 실패하거나 결과가 누락되면 다음 단계로 진행하지 않는 `fail closed` 방식으로 동작한다.
- 승인은 `incident_id`, policy version/digest, generated/canonical command digest, immutable evidence/validation snapshot digest, actor, reason, validation/decision validity window에 결합한다. Dependent policy·command·evidence·analysis·parser·validator·configuration·validation 변경은 승인을 무효화한다.
- v0.1 집행 대상은 격리 컨테이너 또는 네트워크 네임스페이스뿐이다. 호스트·운영 방화벽 집행은 기본 비활성화하며 P0 완료 조건에 포함하지 않는다.
- 적용 rule에는 TTL이 있어야 한다. 새 `execution-result-v2` terminal evidence는 read-back interval에 서명하고 lifecycle은 broad execution interval에서 exact kernel expiry 하나를 추정하지 않고 lower/upper expiry bound를 보존한다. Restart 또는 expiry failure 뒤 recovery는 two-phase journal과 read-back으로 state를 classify하고 expired/unapproved residue만 제거할 수 있으며 missing rule을 re-add하거나 source bound를 재사용하거나 relative TTL을 refresh하지 않는다. Future add는 새 candidate·validation snapshot·exact-artifact HIL decision·single-use capability를 모두 요구한다.
- Gateway는 하나의 고정된 private upstream 앞에 놓이는 비특권 asynchronous data plane이다. `CAP_NET_ADMIN`을 갖지 않고 AI 또는 control plane을 기다리지 않으며, control-plane outage 중 정상 요청을 전달하되 새 block을 생성하거나 적용하지 않는다.

## P0 최종 완료 판정표

아래 항목이 모두 충족되어야 v0.1 릴리스 후보를 완료로 판정한다.

- [ ] Bounded unprivileged Gateway가 fixed private upstream 하나를 proxy하고 direct peer만 identity로 사용하며 forwarding header를 strip/regenerate하고 sanitized `gateway-http-v1` event를 async emit하며 authenticated privacy-minimized `auth-event-v1`을 받고 direct-origin bypass를 막는다 (`FR-022~026`).
- [ ] Header/body/timeout, smuggling/hop-header/streaming/upstream failure, control-plane outage, `500 RPS`/`4 GB`/p95 `5ms` overhead gate가 통과하고 outage 중 valid traffic은 새 block 없이 forward된다 (`NFR-012~014`).
- [ ] 네 탐지 시나리오의 결정론 신호와 교차 소스 incident가 재현된다 (`FR-005~007`).
- [ ] GPT 입력은 compact structured summary이고 출력은 evidence·uncertainty·false positive·single-IP policy·근거 기반 `nft-blacklist-v1` command candidate 하나를 포함한 제한 스키마다 (`FR-008~011`, `FR-021`).
- [ ] 실패·불일치·변조·동시성·직접 내부 호출로 `parse/canonicalize → policy/evidence/command consistency → protected target → nft syntax → historical impact → exact-artifact administrator HIL`을 우회할 수 없다 (`FR-012~013`, `FR-021`, `NFR-001`).
- [ ] Byte-identical HIL 승인 canonical command만 fixed `nft` argument와 stdin을 통해 shell 없이 격리 namespace에서 실행되고 read-back·TTL·expiry·revoke·restart recovery가 작동한다 (`FR-014~016`, `FR-021`).
- [ ] Dashboard가 fact·AI interpretation·generated/canonical command·validation·impact·exact-artifact HIL·apply/expiry·audit를 구분하고 SSE reconnect 및 replay-gap recovery 뒤 수렴한다 (`FR-017~018`, `FR-021`).
- [ ] 정상 traffic과 네 attack command가 Gateway/auth adapter를 반복 통과하고 origin은 private이며 host nftables는 unchanged다 (`FR-019~020`, `FR-022~026`, `NFR-003`, `NFR-011~014`).
- [ ] README의 설치·테스트·데모 명령, 스크린샷·배포 세부, 라이선스, Codex `/feedback` 항목이 실제 결과와 일치한다.
- [ ] Proxy-header spoofing·request smuggling·origin bypass·command injection·digest swap/replay·protected-target mismatch·executor mutation test가 통과하고 저장소·이미지·로그·캡처에 비밀이 없으며 critical P0 defect와 만료되지 않은 demo rule이 0개다.
